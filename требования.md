# Обязательные требования
Семестровая работа – сайт, написанный на Java Spring Boot с использованием Spring MVC Framework. Это должно быть MPA.
Сайт должен иметь удобный интерфейс. Необходимо использовать HTML5, CSS3. CSS-препроцессоры не запрещаются, но нужно знать их основные принципы.
Если используете CSS-фреймворки (Bootstrap и проч.), должны уметь пояснить, что за классы используете, особенно классы сетки. Необходимо корректно работать с формами: проверка данных, защита от повторной отправки там, где это нужно. 
Нужно использовать любой шаблонизатор, можно JSP. В коде видов нельзя напрямую вызывать Java-код.
Должна быть развитая система шаблонов (свои теги или наследование шаблонов, разделение видов на несколько частей).
Сайт должен иметь аутентификацию, авторизацию, регистрацию.
Сайт должен быть защищён от атак, которые обсуждались в течение всего курса предмета(csrf, xss, sql инъекции).
Сайт должен хранить данные в реляционной СУБД PostgreSQL.
Для работы с СУБД необходимо использовать и Spring Data JPA, и JPA. То есть должны быть как и репозитории от Spring, так и самостоятельно написанные запросы с помощью JPQL/CriteriaBuilder.
Система должна работать, как минимум с шестью JPA сущностями. Должны быть связи M2M, O2M.
Как минимум, для одной из сущностей должен быть реализован полностью CRUD: не обязательно в виде админки, но должны быть некие экшены контроллеров, которые реализуют CRUD-действия.
Система должна быть построена по модели MVC, и следует придерживаться логики Controller->Service->Repository(DAO) и т.д.
Необходимо придерживаться принципов SOLID. Весь код должен быть корректно поделен на пакеты. Рекомендуется использовать DTO-, Request-, Forms-классы, где это уменьшает связность, абстрагирует архитектуру.
Проект должен быть основан на Spring Boot: нужно использовать стартеры, настройки и запускающий класс.
Все реализации требований должны вписываться в бизнес-логику и выполнение соответствующего кода должно быть возможно из-за действий пользователя.
Необходимо использовать Docker (сервер приложений и сервер СУБД в отдельных контейнерах).

# Требования к наполнению, наличию реализаций
● Необходимо применить Javascript. Нужно использовать AJAX.
● Все исключения, проблемы должны корректно обрабатываться: обычные экшены – приводить к отдачи самостоятельно оформленной веб-страницы, а AJAX-запросы получить JSON-ответ с информацией о проблеме.
Не должно быть дефолтных страниц с ошибками от сервера приложений или Spring Boot.
● Использование стороннего API, не используя специальную библиотеку для этого API (SDK), а самостоятельное описание запросов на уровне HTTP (например, библиотека okhttp).
Примеры: вход или восстановление пароля через отправку смс, использование курсов валют для пересчёта стоимости корзины.
Наличие простого информера для сайта типа вывода погоды не подходит!
● Использование Spring Security. То есть полноценная работа с пользователем: закрытая часть сайта, аутентификация, регистрация.
● Организация REST API хотя бы для одной сущности. Подключение генерации openAPI. А также написание тестов на выбор через http-файлы Idea, через Postman или аналоги.
● Использование конвертеров, в том числе и своих.
● Логирование возникающих исключений.
● Наличие 2 нестандартных методов для репозиториев Spring:самостоятельно написанных методов с использованием @Query, CriteriaBuiler, но не повторяющих функционал методов, описываемых в Spring Data JPA.
● Один запрос с подзапросом.
● Наличие кэширования с помощью Redis или аналогов.

# Идея сайта и как это реализовано в моем случае

«Культурный навигатор города» — MPA-сайт для поиска культурного досуга в Казани: выставки, концерты, лекции, мастер-классы, музеи, площадки и готовые пользовательские маршруты.

Пользователь может:
смотреть каталог событий и площадок;
фильтровать события по категории, району, датам, времени, цене и рейтингу;
сохранять события в избранное;
оставлять оценки и отзывы;
настраивать предпочтения в профиле;
создавать, редактировать и удалять собственные маршруты;
смотреть публичные маршруты других пользователей.

1. Spring MVC Framework и MPA

Проект реализован как Spring Boot MVC-приложение с многостраничной архитектурой.

Основные страницы:
главная страница `/`;
каталог событий `/events`;
страница события `/events/{id}`;
каталог площадок `/venues`;
страница площадки `/venues/{id}`;
каталог маршрутов `/routes`;
страница маршрута `/routes/{id}`;

личный кабинет `/profile`;
редактирование профиля `/profile/edit`;
страница администрирования `/admin`;
страницы входа и регистрации `/login`, `/register`;
кастомные страницы ошибок `/error`, `400`, `403`, `404`, `500`.

Контроллеры разделены по предметным областям: `HomeController`, `EventController`, `VenueController`, `RouteController`, `ProfileController`, `AdminController`, `AuthController`, а REST-часть вынесена в отдельные `RestController`.

Используется Bootstrap 5, общий файл стилей `app.css` и общий JavaScript.

В интерфейсе реализовано:

личный кабинет с маршрутами, избранным и предпочтениями;
админка с вкладками и быстрым поиском по таблицам;
toast-уведомления для AJAX-действий;

Используется FreeMarker.

Реализованы:

Права:

Реализована защита:

13. Внешнее API без SDK

Используется Yandex Geocoder API без SDK, через обычный HTTP-запрос с OkHttp.

Класс `YandexMapsClient`:
формирует URL `https://geocode-maps.yandex.ru/v1/`;
передает API key, адрес, язык и формат JSON;
выполняет HTTP GET через `OkHttpClient`;
разбирает JSON-ответ через `ObjectMapper`;
извлекает координаты из поля `Point.pos`;
возвращает `GeocodingResult(longitude, latitude)`.

Геокодирование используется:
при сохранении адреса пользователя в профиле;
при создании или редактировании площадки;
для расчета расстояния от пользователя до события;
для расчета примерной длины маршрута между выбранными точками.

Если API-ключ не задан или внешний сервис недоступен, приложение не падает: метод возвращает `Optional.empty()`, ошибка логируется, а координаты просто не обновляются.

17. Два нестандартных метода репозитория

Через `@Query` реализованы нестандартные JPQL-запросы:

Он подбирает публичные маршруты по диапазону бюджета, максимальной длительности и минимальному количеству точек.

Подзапросы используются в JPQL.

21. Как реализован drag-and-drop маршрута

Drag-and-drop реализован на стороне клиента в `app.js` в функции `initRouteBuilder()`.

Как это работает:
в шаблоне формы маршрута каждая выбранная точка маршрута рендерится как `.route-selected-item`;
для выбранных элементов задан `draggable="true"`;
при начале перетаскивания срабатывает `dragstart`, элемент получает класс `is-dragging`;
во время `dragover` определяется, перед каким элементом нужно вставить перетаскиваемую точку;
для этого используется функция `getDragAfterElement(container, y)`;
она сравнивает координату курсора `clientY` с вертикальным центром соседних элементов;
если курсор выше центра элемента, текущая точка вставляется перед ним;
если подходящего элемента нет, точка добавляется в конец списка;
после `dragend` класс `is-dragging` снимается, а сводка маршрута пересчитывается.

Порядок выбранных событий сохраняется не в отдельном JavaScript-массиве, а прямо в DOM: у каждой точки есть скрытый input `name="eventIds"`. При отправке формы браузер отправляет `eventIds` в текущем порядке DOM-элементов, поэтому сервер получает уже упорядоченный список.

Для пользователей, которым неудобно перетаскивание, оставлены кнопки «Вверх», «Вниз», «Убрать». Они меняют тот же DOM-порядок.

22. Как работает расчет расстояний

Расчет расстояний есть на сервере и на клиенте.

На сервере:
адрес площадки геокодируется через Yandex Geocoder API;
координаты сохраняются в `Venue.latitude` и `Venue.longitude`;
адрес пользователя из профиля тоже геокодируется и сохраняется в `User.preferredLatitude` и `User.preferredLongitude`;
для расстояния от пользователя до события используется метод `YandexMapsClient.distanceKm(...)`;
для длины маршрута используется `YandexMapsClient.estimateDistanceKm(...)`;
метод проходит по точкам маршрута по порядку и складывает расстояния между соседними точками.

Формула:
используется haversine-формула;
она считает расстояние между двумя координатами на сфере;
радиус Земли принят как 6371 км;
результат округляется до одного знака после запятой.

На клиенте:
в форме маршрута у каждого события есть data-атрибуты с координатами площадки: `data-lat`, `data-lon`;
после добавления, удаления или перетаскивания точки вызывается пересчет сводки;
клиентская функция `estimateRouteDistance(items)` берет выбранные точки в текущем порядке и тоже считает haversine-расстояние;
это дает пользователю быстрый предварительный расчет до отправки формы;
окончательное значение все равно пересчитывается на сервере при сохранении маршрута.

23. Как работает живая сводка маршрута

В форме маршрута есть блок `data-route-summary`.

Он показывает:
количество выбранных точек;
общую стоимость билетов;
запланированную длительность;
примерную дистанцию;
подсказку по бюджету.

Стоимость берется из `data-price` выбранных событий.
Длительность берется из поля `durationMinutes`.
Бюджет берется из поля `budget`.
Если сумма билетов выше заявленного бюджета, пользователь сразу видит предупреждение.

24. Как работают AJAX-отзывы

На странице события есть контейнер `data-reviews-list`.

При загрузке страницы JavaScript вызывает:
`GET /api/events/{eventId}/reviews`.

Ответ JSON рендерится в список отзывов без перезагрузки страницы.

При отправке формы:
JavaScript собирает оценку и текст;
отправляет `POST /api/events/{eventId}/reviews`;
передает CSRF-токен в заголовке;
блокирует кнопку на время запроса;
после успешного ответа очищает форму;
повторно загружает список отзывов;
показывает сообщение «Отзыв сохранён».

Если запрос завершился ошибкой, пользователь видит понятное сообщение рядом с формой и/или toast-уведомление.

25. Как работают карточки и бейджи

Карточки событий и маршрутов являются кликабельными целиком через атрибут `data-card-href`.
Внутренние ссылки с названий событий, маршрутов и площадок внутри карточек убраны, чтобы не было двойной интерактивности и визуального шума.

Бейджи событий считаются по DTO `EventCardDto`:
`Сегодня` — если дата начала события совпадает с текущей датой;
`Бесплатно` — если цена равна 0;
`Рядом` — если расстояние до пользователя не больше 5 км;
`Высокий рейтинг` — если средняя оценка не ниже 4.5.

Карточки, полученные через AJAX-фильтрацию, рендерятся теми же бейджами в `app.js`, чтобы внешний вид не отличался от серверного рендера.
